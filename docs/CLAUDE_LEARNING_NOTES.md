## 1. 모드 & 단축키

### Q. Auto 모드 안 쓰는 게 좋나? Plan 모드는 정확히 뭐?
**Auto 모드** (Auto-accept edits) = 권한 안 묻고 파일 수정 자동 수락.

| 상황 | Auto 권장? |
|---|---|
| 새 기능, 복잡 리팩토링 | ❌ — 엉뚱한 파일 건드리면 골치 |
| 이름 일괄 변경, 포맷팅 | ✅ — 매번 묻는 게 더 짜증 |
| 처음 만지는 코드 | ❌ — 확인하면서 |
| 검증 명령 + 커밋 잘 됨 | ✅ — 안전망 있음 |

**Plan 모드** = 계획만 짜고 코드 절대 안 건드림.
- ✅ 가능: 파일 읽기, grep, 명령 실행(읽기), 분석
- ❌ 차단: 파일 수정·생성, 파괴적 명령

**흐름**: Shift+Tab → Plan → 계획 받기 → "왜?" 반박 → 확정되면 Exit → 코드 작성

**왜**: 바로 코드 짜면 잘못 이해한 채로 200줄 만들고 다 버리게 됨.

### Q. Bash 커맨드가 뭔가요?
**터미널에서 치는 명령어** = Bash 명령어.

```bash
git status          # 변경사항 확인
git commit -m "..."  # 커밋
python -m http.server 8000  # 서버 띄우기
docker compose up   # 도커 실행
ls / cd             # 폴더 보기/이동
```

**왜 "Bash"라고 부르나**: 리눅스·맥의 기본 명령어 해석기 이름. Windows는 PowerShell이 기본이지만 Claude Code는 둘 다 지원.

**settings.json에서 표기 방식**:
```json
"Bash(git status:*)"
   ↑       ↑      ↑
 도구    명령어  뒤에 뭐 붙어도 OK
```

| 표기 | 의미 |
|---|---|
| `Bash(git status)` | 딱 `git status`만 |
| `Bash(git status:*)` | `git status -s` 등 뒤에 뭐 붙어도 |
| `Bash(rm -rf:*)` | `rm -rf` 시작하는 모든 거 |

**Claude Code의 도구 종류**: Bash, PowerShell(둘 다 터미널), Read(읽기), Edit(수정), Write(새 파일).

## 2. 메모리·룰 운영

### Q. 메모리는 "감독관 모듈" 같은 건가요?
"감독관"이 뭘 가리키냐에 따라 다름:

**해석 1: 룰·매뉴얼이라면** → 강제냐 참고냐 차이

| 항목 | CLAUDE.md (룰) | 메모리 (노트) |
|---|---|---|
| 비유 | 회사 사규 — 강제 | 업무 노트 — 참고 |
| 로드 | 매 세션 자동 + 강제 | 관련 있을 때만 |
| 내용 | "해라/하지 마라" | "이걸 알아두면 도움" |

**한 줄**: CLAUDE.md는 "법", 메모리는 "일기".

---

### Q. "작업을 담당해서 관찰하는 역할"은? (서브에이전트/Hook)
메모리·매뉴얼이 죽은 문서라면, **서브에이전트**와 **Hook**은 살아있는 모듈.

| 항목 | 메모리 | 서브에이전트 | Hook |
|---|---|---|---|
| 성격 | 죽은 문서 | 살아있는 일꾼 | 자동 반응 |
| 동작 | 펴봐야 작동 | 작업 직접 실행 | 이벤트 발동 |
| 비유 | 일기장 | 보조 직원 | CCTV + 경보 |
| 예시 | "어제 RLS 실수" | "코드리뷰 전담" | "저장하면 자동 lint" |

**서브에이전트 예** (본 프로젝트):
- `migration-reviewer` — 마이그레이션 작성되면 RLS·트랜잭션·검증 SELECT 자동 체크
- `frontend-tester` — HTML/JS 변경 시 브라우저로 자동 검증
- `grading-validator` — grading-server 변경 시 import·라우터 검증

**Hook 예**:
- 코드 짜기 끝 → 자동 `ruff check`
- 커밋 직전 → 자동 `pytest`
- Edit 호출됨 → 변경 파일 자동 백업

**도입 순서**: 매뉴얼·메모리(1단계) → 서브에이전트 1개 → Hook 자동화.

---

### Q. 메모리에 저장해서 나중에 문제 이어서 작업?
**부분적으로 맞지만, 메모리는 "할일 목록"이 아니라 "맥락"**.

**좋은 메모리 vs 안 맞는 것**:

| ✅ 좋음 | ❌ 안 맞음 |
|---|---|
| "RLS 래핑하면 망함" (재발 방지) | "버튼 색 빨강으로" (TODO) |
| "SQL Editor에서만 실행" (특이 규칙) | "오늘 회의 3시" (일정) |
| "Mathpix 폐기" (과거 결정) | "함수명 changeColor" (grep) |
| "1인 개발자" (선호도) | "Phase 5a 완료" ⚠️ 시간 지나면 거짓 |

**"나중에 이어서 작업" 용도라면**:

| 목적 | 도구 |
|---|---|
| "이거 나중에 함" | TODO 주석, Issues |
| "어제 일 이어" | `/resume` |
| "복잡 작업 단계" | TodoWrite (세션 내) |
| "다음 세션이 알아야 할 사실" | **메모리** |

## 3. 세션 관리

### Q. `/memory` 입력하니 "Continue in Terminal to edit memory?" 뜸. 뭔가요?
VSCode 확장에서는 메모리 **편집 UI**를 띄울 수 없어서 터미널로 가야 한다는 안내.

| 선택 | 동작 | 언제 |
|---|---|---|
| 1. Continue in Terminal | 터미널 창에서 편집 UI | 메모리 직접 수정·삭제 |
| 2. Never mind | 취소 | 그냥 보고 싶었을 뿐 |

**메모리 보기만 하려면**: 그냥 "지금 적용된 메모리 보여줘"라고 물어보거나, `~/.claude/projects/.../memory/` 폴더 직접 열기.

---

### Q. 권한·룰 바꾸면 클로드 껐다 켜야 적용?
**네, 깔끔하게는 재시작.**

| 변경한 것 | 적용 시점 |
|---|---|
| `settings.json` 권한 | 새 세션부터 |
| `CLAUDE.md` 룰 | 새 세션 시작 시 자동 로드 |

현재 세션은 옛 상태로 돌고 있어서 재시작해야 진짜 효과.

**방법**: 창 닫고 다시 열기 → 새 세션에서 `/memory` 또는 `/context`로 확인.

---

### Q. 이 창 두고 다른 창에서 해도 됨?
**네, 오히려 안전.**

1. 이 창은 그대로 둠 (작업 내역 보존)
2. 새 창 띄우기 (같은 프로젝트)
3. 새 창에서 `/memory` 확인 → 룰 잘 로드됐는지
4. 실제 작업도 새 창에서

이미지 TIP 41(멀티 클로드)이 이거.

### Q. 이 창은 작업방식 물어보면서 쓰고 싶음
**좋은 활용 — 코치/작업 분리.**

| 창 | 역할 | 사용법 |
|---|---|---|
| 이 창 (코치) | 자문 | "이거 어떻게 해?", "이 명령어 뭐야?" 질문만 |
| 새 창 (작업) | 실제 작업 | Plan → 코드 → 검증 → 커밋 |

**주의**: 코치 창에선 코드 수정 시키지 말기 (옛 권한 상태라 충돌).

**자주 물을 만한 것**:
- "방금 새 창에서 이런 에러 나왔는데?"
- "Shift+Tab 안 먹어"
- "MCP 끊겼다는데?"
- "이번 작업은 Opus야 Sonnet이야?"

## 4. 권한 시스템

### Q. "적용 상태" 표 (글로벌·프로젝트 settings, deny 등록)가 뭔가요?
권한 화이트리스트 = **자동문 출입증**.

| 분류 | 의미 |
|---|---|
| **allow** | 묻지 말고 바로 실행 — 출입증 직원 |
| **deny** | 절대 실행 금지 — 출입금지 구역 |
| 둘 다 아님 | 매번 본인에게 확인 |

**두 위치 분리 이유**:

| 파일 | 범위 | 예시 |
|---|---|---|
| `~/.claude/settings.json` (글로벌) | 전 프로젝트 | `git`, `pytest`, `npm` |
| `.claude/settings.local.json` (프로젝트) | 본 프로젝트만 | `docker`, `uvicorn` |

**구체 내용**:

✅ 자동 허용:
- `git status/diff/log/commit/push`
- `pytest`, `ruff`, `mypy`, `python -m`
- `npm run/install`
- `docker compose ps/up` (프로젝트)
- `uvicorn` (프로젝트)

❌ 자동 차단:
- `rm -rf`, `git push --force`, `git reset --hard`
- `sudo`, `chmod 777`, `curl ... | sh`
- `supabase apply_migration` (SQL Editor 직접 패턴 위반)

**체감 변화**:
```
[전] git status 실행해? [Y/n]   ← 매번 묻기
[후] git status → 자동 ✅
     rm -rf project → 차단 ❌  (Y 눌러도 안 됨)
```

YOLO 모드(`--dangerously-skip-permissions`) 켜도 **deny는 여전히 막힘** — 그래서 안전.

## 5. 워크플로우

### Q. "끝나면 노트에 저장" / "검수 단계" 쉽게 정리해줘

**노트 저장 (Step 4)**
- 왜: 클로드는 매일 기억 리셋. 안 적어두면 내일 또 같은 실수.
- 뭘: **"몰랐다가 망했던 것"**만. (✅ 실패 사례·특이 규칙 / ❌ 코드 보면 보이는 것)
- 어떻게: "오늘 알아낸 거 메모리에 저장해줘"
- 다음날: 새 창 켜면 자동 로드. 또는 `/resume`.
- 주의: 메모리는 어제 시점 스냅샷. 현재 코드 한 번 더 확인.

**검수 (Step 2)**
- 왜: 코드 짰다고 끝 아님. 작동 확인 안 하면 90% 망함.
- 뭘로:
  - 메인 사이트 → 브라우저 클릭 + F12 콘솔 에러 0
  - 채점 서버 → `python -c "from main import app"`
  - Docker → `docker compose up --build`
  - 마이그레이션 → SQL Editor 직접 (MCP 금지)
- 클로드 감시: thinking block 보다가 이상하면 즉시 `ESC`.

**한 줄**: 코드 짜기 끝 ≠ 끝. **돌려보고 → 이상하면 멈추고 → 끝나면 기억 저장.**

## 6. 용어·약어

### Q. Spike가 뭔가요? ("5/3에 $8 튀고 나머지는 $1 미만")
**Spike = 갑자기 튀어오른 값**. 비용·트래픽·CPU 등이 평상시보다 급증한 순간.

비유: 심전도 그래프에서 한 곳만 뾰족 솟은 부분.

```
$8 ┤            ▲  ← spike (5/3)
$1 ┤▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁
    1 2 3 4 5 6 7 ...일
```

**"평소 일평균 — spike 제외하고"** = 5/3의 $8을 빼고 평범한 날들 평균. Spike 포함하면 평균이 왜곡되니까.

**본 프로젝트의 spike 발생처**:

| 어디 | 원인 |
|---|---|
| Gemini Vision 비용 | 큰 PDF 한 번에 처리 |
| Supabase 요청 | RLS 회귀로 무한 재시도 |
| Drive API | 숙제 제출 폭주 |
| CPU·메모리 | OCR 동시 처리 |

**유사 용어**:

| 용어 | 뜻 |
|---|---|
| **spike** | 갑자기 튀어오른 값 (점) |
| **outlier** | 이상치 (전반에서 벗어난 값) |
| **baseline** | 정상 상태 기준선 |
| **anomaly** | 비정상 패턴 (spike보다 넓음) |
| **p95 / p99** | 상위 5%·1% 값 (spike 포함된 끝값) |

### Q. 마이그레이션이 뭔가요?
**DB 구조를 단계적으로 바꾸는 SQL 파일.**

영어로 "이주·이동". 본 프로젝트에선 거의 항상 **Supabase 테이블 구조 변경**을 가리킴.

**비유**: 가게 인테리어 변경 작업 지시서. 테이블 추가, 컬럼 추가, 인덱스 추가 등.

**파일 예**:
```sql
-- migrations/0030_add_phone_20260513.sql
BEGIN;
ALTER TABLE students ADD COLUMN phone TEXT;
SELECT count(*) FROM students WHERE phone IS NULL;
COMMIT;
```

**왜 파일로 남기나**: DB 구조가 계속 진화 → 각 변경을 파일로 적어두면 Git commit처럼 추적 가능.

**본 프로젝트 규칙** (메모리·CLAUDE.md에 박힘):

| 규칙 | 이유 |
|---|---|
| `migrations/NNNN_*_YYYYMMDD.sql` 명명 | 순서·날짜 |
| `BEGIN; ... COMMIT;` 감싸기 | 실패 시 롤백 |
| 하단에 검증 SELECT | 적용 결과 확인 |
| **SQL Editor 직접 붙여 실행** | MCP `apply_migration` 금지 (회귀 사례) |

**트리거 단어**: "**마이그레이션 만들어줘**" → 템플릿 자동 작성 + SQL Editor 안내.

**다른 종류 마이그레이션** (참고용):

| 종류 | 의미 |
|---|---|
| DB 마이그레이션 | 테이블 구조 (← 본 프로젝트) |
| 코드 마이그레이션 | 라이브러리 교체 (예: jQuery → React) |
| 서버 마이그레이션 | 클라우드·인프라 이전 |
| 데이터 마이그레이션 | DB → DB 데이터 이동 |

---

### Q. 백엔드 in-flight 차단 / fallback 단순화 / 429 backoff / Drive sync 폴링이 뭔가요?
**4개가 사실 spike 재발 방지 4종 세트**. 따로 보면 어렵지만 묶으면 명확.

#### 백엔드 in-flight 차단
- `in-flight` = 비행 중 → **"현재 처리 중인 작업"**
- 같은 요청이 동시에 여러 번 들어오면 첫 번째만 처리, 나머지 거절
- **비유**: 엘리베이터 버튼 — 여러 번 눌러도 한 번만 감
- **왜**: 사용자가 "분석" 버튼 연타하면 API 비용 5배. spike 사고 원인.

#### 모델 fallback 단순화
- `fallback` = 차선책
- 메인 실패 시 다른 모델로 → 체인이 너무 길면 비용·시간 폭발
- **비유**: 1번 식당 닫혀서 2번→3번→4번→5번 다 돌면 기름값 폭발. **2곳만 시도하고 포기**가 나음
- **왜**: spike 사고 원인 — `OCR×3모델 + 풀이×2모델` 체인이 5만원 터짐

#### 429 backoff
- `429` = HTTP "Too Many Requests" 에러 코드
- `backoff` = 물러나서 대기 (1초→2초→4초→8초 지수 증가)
- **비유**: 콜센터 통화 중이면 1분 기다리고 재시도, 또 통화 중이면 2분 기다림
- **왜**: API 분당 호출 한도 넘으면 즉시 재시도 X, 기다려야 함

#### Drive sync 폴링 패턴 (3초 간격, elapsed 표시)
- `폴링(polling)` = 주기적으로 "다 됐어?" 물어보기
- `3초 간격` + `elapsed 표시` = 3초마다 확인 + "5초 경과", "8초 경과" 화면에 노출
- **비유**: 배달 앱 — 3초마다 상태 새로고침 + "주문 후 12분 경과" 표시
- **왜**: 업로드 중 화면이 멈춘 듯 보이면 사용자가 불안해서 버튼 또 누름 → in-flight 사고. 진행 표시로 연타 자체 차단.

#### 묶음 작동 흐름
```
사용자 "처리 중인지 몰라 연타"
   ↓ 폴링으로 진행 표시 (④) → 연타 자체 차단
   ↓ in-flight 차단 (①) → 중복 요청 무시
   ↓ fallback 단순화 (②) → 비용 폭탄 방지
   ↓ 429 backoff (③) → API 한도 부드럽게 회피
```

= **2026-05-09 spike 사고 재발 방지 4종 세트**. 메모리 `feedback_safety_guards_for_automation` 참조.

### Q. 스모크 테스트·E2E 테스트가 뭔가요? 다른 테스트는?

#### 핵심 2개
**Smoke test (스모크)** = "전원 켜지냐?" 최소 기본 확인.
- 어원: 전자제품 켰을 때 **연기 안 나면 OK**
- 1~5분 안에 끝남. "서버 뜨나? 로그인 화면 보이나?"
- 본 프로젝트 예: `python -c "from main import app"`, `curl http://localhost:8000/`

**E2E test (End-to-End)** = 사용자 행동 처음~끝까지 따라하기.
- "학원 손님처럼 로그인 → 출석 → 결제까지 통째로"
- 본 프로젝트 예: 학부모 로그인 → 자녀 출석 확인 → 수납 보기
- 도구: Playwright, Cypress, Selenium

#### 전체 종류

| 테스트 | 뜻 | 비유 | 속도 |
|---|---|---|---|
| **Unit (단위)** | 함수 1개 검증 | 부품 하나 검사 | 빠름 (ms) |
| **Integration (통합)** | 여러 모듈 연동 | 부품 조립 후 맞나 | 중간 (초) |
| **E2E** | 전체 흐름 | 실제 주행 | 느림 (분) |
| **Smoke** | 최소 기본 | 시동 걸리나? | 빠름 |
| **Regression (회귀)** | 옛 기능 안 망가졌나 | 어제 되던 게 오늘도? | 다양 |
| **Acceptance (수락)** | 요구사항 충족 | 고객 만족? | 느림 |
| **Performance (성능)** | 속도·응답 | 100m 몇 초? | 다양 |
| **Load (부하)** | 동시 사용자 견디나 | 100명 한꺼번에? | 느림 |
| **Stress** | 한계까지 밀기 | 어디서 터지나? | 느림 |
| **Snapshot** | 화면 그대로? | 어제 사진과 비교 | 빠름 |

#### 테스트 피라미드 (실무 정석)
```
       ╱E2E╲           ← 적게 (느리고 비쌈)
      ╱─────╲
     ╱통합   ╲          ← 중간
    ╱─────────╲
   ╱  단위     ╲        ← 많이 (빠르고 쌈)
```

**단위는 많이, E2E는 핵심 흐름만** = 가성비.

#### 본 프로젝트 현실
| 영역 | 검증 방식 |
|---|---|
| 프론트 (Vanilla JS) | 정식 X, 브라우저 직접 클릭(수동 E2E) |
| 백엔드 (Python) | `pytest` 가능 (권한 있음) |
| 채점 파이프라인 | `vision-diagnose` 엔드포인트(스모크+통합) |
| Supabase | SQL Editor 검증 SELECT |

**부족**: 자동 E2E. **추천 도입 순서**: 스모크 → 회귀 → E2E.

### Q. Alert이 뭔가요?
**2가지 의미** — 어디서 봤냐에 따라 다름.

#### ① JavaScript `alert()` — 브라우저 팝업
```javascript
alert("저장되었습니다");
```
- "확인" 눌러야 다음 행동 가능 (차단형)
- **비유**: 길에서 "잠깐요!" 막아세우기
- 본 프로젝트의 `index.html`·`script.js`에 "저장됨" 같은 알림으로 박힘
- 트렌드: 못생기고 차단형이라 `toast`·`modal`로 대체 중

#### ② 모니터링 Alert — 시스템 이상 자동 알림
- 도구가 감시 → 이상 감지 시 본인에게 Slack·이메일·SMS
- 예: "Gemini API 비용 1시간 $10 넘음", "서버 응답 5초 초과"
- **비유**: 화재경보기 — 평소 조용, 연기 나면 삑삑
- 도구: Sentry, Datadog, Grafana, UptimeRobot
- 본 프로젝트 연결점: 메모리의 "자동 작업 안전 가드 3종" 중 **실패 모니터링**이 정확히 이거. spike 사고 막으려면 필요.

#### 비교

| 종류 | 누구한테 | 언제 | 어떻게 |
|---|---|---|---|
| `alert()` 팝업 | **사용자**에게 | 액션 결과 즉시 | 브라우저 차단 팝업 |
| 모니터링 alert | **개발자(본인)**에게 | 시스템 이상 시 | Slack/이메일/SMS |

**한 줄**: 사용자한테 보여주면 `alert()`, 본인이 사고 막으려고 받으면 모니터링 alert.

### Q. "주기적으로 계속 폴링 (idle 상태에도) — 하루 종일 페이지 열어둔 경우 자동 갱신" 옵션이 뭔가요?
**"페이지만 열어둬도 자동으로 새로고침할지" 결정하는 체크박스.**

#### 용어
- **폴링** = 3초마다 "새로운 거 있어?" 물어보기
- **idle 상태** = 한가한 상태 (아무 작업 안 함)

#### 동작 비교
| 모드 | 동작 | 장단점 |
|---|---|---|
| OFF (현재 기본) | 작업 진행 중일 때만 폴링, 끝나면 정지 | 서버·비용 적음 / 새로고침 본인이 해야 함 |
| ON (체크 시) | 페이지 열린 동안 계속 폴링 | 항상 최신 / **API 비용·서버 부담 누적** |

#### 비유
- OFF: 배달 시켰을 때만 앱 새로고침
- ON: 하루 종일 3초마다 배달 앱 새로고침

#### ⚠️ 본 프로젝트 권장
**OFF 유지**. spike 사고($5만, 2026-05-09) 이력 있음.
- 직원 8시간 × 1200회 × 인원수 = 비용 누적
- 정말 필요하면 폴링 간격 늘리거나(30초+), idle 1시간 후 자동 정지 필요

### Q. "본인 PC CLI (추천)"이 뭔가요?
**본인 컴퓨터의 터미널 창에서 명령어로 직접 실행하는 방식.**

#### CLI = Command Line Interface
검은 터미널 화면에서 명령어로 쓰는 프로그램. 본 프로젝트에서 자주 쓰는 것들:
```bash
python -m http.server 8000
docker compose up
git status
```

#### 3가지 실행 위치 비교
| 방식 | 어디서 | 장점 | 단점 |
|---|---|---|---|
| **본인 PC CLI** ⭐ | 본인 PC 터미널 | 비용 0, 디버깅 쉬움, 빠른 반복 | PC 꺼지면 멈춤 |
| **클라우드** (Railway 등) | 외부 서버 | 24시간, PC 안 켜둬도 됨 | 비용, 환경 설정 까다로움 |
| **웹 UI 버튼** | 브라우저 | 비전공자 쉬움 | 옵션 제어 제한 |

#### 비유
- 본인 PC CLI = 본인이 직접 청소
- 클라우드 = 청소업체 정기 출장
- 웹 UI = 로봇청소기 버튼

#### 왜 추천인가
- 비용 0
- 에러 즉시 보임 → 디버깅 쉬움
- 코드 고치고 바로 재실행
- 시크릿은 `.env.local`에 두면 끝
- 본 프로젝트는 `settings.local.json`에 `python`/`docker compose` 등 권한 이미 있음

#### 클라우드로 옮겨야 할 신호
- 1시간 넘는 장기 작업
- 새벽·정해진 시간 자동 실행
- 여러 명 트리거
- PC·인터넷 끊겨도 가동
- (예: 본 프로젝트 해설제작 RAG Phase 2 — Railway 자동 빌드)

**한 줄**: 일회성·디버깅·빠른 반복 = 본인 PC CLI / 정기·자동·24시간 = 클라우드.

## 🎯 시나리오 — 토의 자동 진입 + 페르소나 격리 (2026-05-18 정립)

> 오늘 시스템 정비로 결정된 표준 흐름. 외워둘 가치 충분.

### 핵심 룰 3종
1. **토의 시작 = plan mode 자동 진입** — "토의해주세요" 한마디로 Claude가 `EnterPlanMode` 호출. Shift+Tab 불필요.
2. **페르소나는 작업창별 격리** — 한 작업창에서 14개 (글로벌 12 + 그 프로젝트 2). 해설 토의는 해설 작업창, 매니저 토의는 매니저 작업창.

### 표준 시나리오 — "해설지 OCR 실패 시 fallback 추가하고 싶다"

```
[1] 사용자: 해설 작업창 열기 (시험지 해설 제작/highroad-math-solution/)
    ↓ ⚠️ 매니저 작업창 X — haeseol-* 페르소나 호출 안 됨

    ↓
[3] Claude: EnterPlanMode 자동 호출 → 화면 상단 "Plan mode" 표시
    ↓ ⚠️ 사용자: mode 표시 1초 체크. 없으면 "plan mode 진입 안 했네, 다시"

    ↓

    ↓
[6] Claude: chief-reviewer 종합 → Plan 작성 → ExitPlanMode (사용자 승인)
    ↓
[7] 사용자: Plan 승인 → 일반 모드 복귀 → haeseol-developer 구현 시작
    ↓
[8] 검증 (Step 2 절차) → commit + push (Step 3 절차)
```

### 잘못된 시나리오 (회피)

❌ commit 후 그냥 `git push`
